# 贪吃蛇（Pygame）个人学习项目

这是一个个人学习项目，使用 Pygame 内置绘图能力实现，**不依赖外部字体、图片或音频资源**。
本 README 同时作为架构设计文档和学习路径指南，不包含代码实现。

## 1. 项目目标

开发一个基于 Pygame 的桌面端贪吃蛇小游戏，具备完整 GUI 与可扩展能力，核心功能包括：

- 难度选择（至少简单 / 普通 / 困难）
- 排行榜（本地持久化、可按难度区分）
- 完整页面流转（主菜单、游戏中、暂停、结算、排行榜、设置）
- 友好的交互反馈（动画、提示音、按钮状态）

## 2. 功能范围

### 2.1 必备功能

- 主菜单
  - 开始游戏
  - 查看排行榜
  - 设置
  - 退出
- 难度系统
  - 不同速度
  - 可选不同地图规则（是否有墙体穿越）
  - 可选不同生成策略（食物价值、障碍出现频率）
- 游戏核心
  - 蛇移动、转向、防反向
  - 食物生成与得分
  - 撞墙/撞自身判定
  - 暂停与继续
- 结算系统
  - 展示分数、时长、吃食物数量
  - 输入昵称并提交成绩
- 排行榜
  - 按难度分榜
  - 显示前 N 名
  - 支持清空（可加二次确认）

### 2.2 增强功能（个人项目暂不实现）

以下功能在核心玩法稳定后可自行探索，初期学习阶段跳过即可：

- 音乐与音效开关（需要外部音频文件，暂不实现）
- 主题皮肤（浅色/深色）（暂不实现）
- 连击或加分机制
- 成就系统（如“单局 100 分”）

## 3. 总体架构

采用“分层 + 状态机”架构，降低耦合、便于扩展。

### 3.1 分层说明

- 表现层（UI Layer）
  - 负责页面绘制、按钮与文本输入、交互反馈
- 领域层（Domain Layer）
  - 负责游戏规则、碰撞判定、计分逻辑、难度策略
- 应用层（Application Layer）
  - 负责页面状态切换、流程编排、事件分发
- 基础设施层（Infrastructure Layer）
  - 负责本地数据存储和配置加载；绘制全部使用 pygame 内置 API，无外部资源依赖

### 3.2 运行机制

- 单主循环驱动
  - 输入处理
  - 逻辑更新
  - 场景渲染
- 使用固定节拍或基于时间差更新，保证不同机器上的一致体验
- 页面切换由统一状态机管理，避免分散跳转逻辑

## 4. 建议目录结构

个人项目无外部资源，结构保持精简：

- `data/`
  - 排行榜与本地配置存档（JSON 文件）
- `src/`
  - `app/` 应用启动与主循环入口
  - `core/` 游戏核心模型与规则
  - `ui/` 页面、控件
  - `services/` 排行榜、配置等服务
  - `states/` 场景状态机与页面状态定义
  - `utils/` 通用工具（常量、校验）
- `tests/`
  - 规则层与服务层测试
- `README.md`

## 5. 核心模块设计

### 5.1 场景状态机模块

负责统一管理页面状态：

- `MENU` 主菜单
- `DIFFICULTY_SELECT` 难度选择
- `PLAYING` 游戏中
- `PAUSED` 暂停
- `GAME_OVER` 结算
- `LEADERBOARD` 排行榜
- `SETTINGS` 设置

状态切换由应用层触发，页面本身只关心展示与输入，不直接修改底层数据。

### 5.2 游戏核心模块

核心对象：

- 蛇实体：位置列表、当前方向、下一方向、长度变化
- 食物实体：坐标、分值、刷新规则
- 地图实体：网格大小、边界规则、障碍信息
- 局内会话：分数、时长、难度、状态

关键规则：

- 输入防抖与防反向
- 碰撞检测优先级
- 食物刷新需避开蛇身与障碍
- 速度曲线由难度策略决定

### 5.3 UI 模块

建议将 UI 分为“页面 + 组件”两级：

- 页面：主菜单页、游戏页、暂停页、结算页、排行榜页、设置页
- 组件：按钮、标签、输入框、提示弹窗、表格行

UI 只发出“用户意图事件”（如开始游戏、提交成绩），由应用层解释并执行。

### 5.4 排行榜模块

职责：

- 成绩记录写入
- 按难度分组查询
- 排名规则排序
- 截断保留前 N 条

建议字段：

- 玩家昵称
- 分数
- 难度
- 局内时长
- 时间戳

排序建议：

- 主排序：分数降序
- 次排序：用时升序
- 再次排序：时间戳升序（先达成者优先）

### 5.5 配置模块

配置内容建议包含：

- 窗口大小与帧率
- 默认难度
- 颜色常量（蛇体色、食物色、背景色等）
- 字体使用 pygame 内置系统字体，无需外部字体文件

> 本项目不使用外部图片、音频或字体资源，所有视觉元素均通过 pygame 的 `draw` 和 `font` 模块实现。

## 6. 难度系统设计

### 6.1 难度维度

- 移动速度（每秒步数）
- 地图复杂度（障碍数量）
- 奖励密度（高分食物出现概率）
- 容错机制（是否允许穿墙）

### 6.2 推荐方案

- 简单：低速、无障碍、可选穿墙
- 普通：中速、少量障碍、标准规则
- 困难：高速、较多障碍、无穿墙

采用“策略配置化”方式，将难度参数集中管理，后续可新增“地狱模式”而无需改动核心流程。

## 7. 页面流程设计

主流程：

1. 启动进入主菜单
2. 进入难度选择
3. 开始游戏
4. 游戏结束进入结算
5. 保存成绩
6. 返回主菜单或查看排行榜

中断流程：

- 游戏中按暂停键进入暂停页
- 暂停页可继续/重开/回主菜单

异常流程：

- 排行榜数据损坏时回退到空榜并记录错误

## 8. 数据设计

### 8.1 本地存储对象

- 配置数据
  - 音量、主题、默认难度、按键偏好
- 排行榜数据
  - 按难度分组的成绩集合

### 8.2 数据约束

- 昵称长度限制
- 分数非负
- 难度值必须在定义集合中
- 读取失败时兜底默认值

### 8.3 持久化策略

- 启动时读取
- 提交成绩时写入
- 设置变更时即时写入
- 写入采用“先生成新内容再覆盖”降低损坏风险

## 9. 事件与输入设计

### 9.1 输入分层

- 系统事件：退出、窗口变化
- 全局热键：暂停、静音
- 页面事件：按钮点击、输入框提交
- 游戏事件：方向输入、重开输入

### 9.2 关键约束

- 单帧只接受一次方向变更
- 禁止从左直接到右、从上直接到下
- 文本输入需过滤非法字符

## 10. 可测试性设计

优先对“规则稳定、纯逻辑”的模块做测试：

- 碰撞判定
- 食物生成合法性
- 计分规则
- 排行榜排序与截断
- 数据读写容错

UI 层以手工回归为主，重点覆盖页面跳转与输入交互。

## 11. 性能与体验指标

- 常规设备下稳定帧率
- 页面切换无明显卡顿
- 资源加载失败不致崩溃
- 单局结束到提交成绩的操作路径简短

## 12. 开发里程碑（建议）

### 里程碑 1：基础骨架

- 完成窗口与主循环
- 完成状态机与页面路由
- 可从主菜单进入空白游戏页并返回

### 里程碑 2：核心玩法

- 实现蛇移动、食物、碰撞、得分
- 接入难度配置
- 完成结算页

### 里程碑 3：排行榜与设置

- 本地成绩持久化
- 排行榜页面与按难度筛选
- 设置页（音量、主题、默认难度）

### 里程碑 4：打磨与验收

- 动效与提示优化
- 异常处理完善
- 测试与文档补齐

## 13. 验收标准

- 能完整完成“主菜单 → 选难度 → 对局 → 结算 → 入榜”闭环
- 三档难度体感明显不同
- 排行榜正确排序且稳定保存
- 关键异常有兜底策略
- 文档与目录结构一致，便于后续继续开发
